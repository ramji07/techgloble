(1) Transformation of an E-R Schema to Table :

Reduction or transformation of E-R schema to table means conversion of E-R design to relation design. This can be done because both the E-R model and Relation Database Model are abstract, logical representation of real world enterprise and the two models empliy similar design principles.

A E-R database schema can be represented by collection of tables, which is knows as Relation Database Schema. Each entity set and relationship set are converted into tables with unique name. All the attribute of the entity sets & relationship sets are represented by multiple column of the tables.

Although there is no particular procedure specified or followed for the transformation of E- R model to relational database model but the whole process can be divided in following seven logical steps -

1. Transformation of strong Entity sets.

2. Transformation of Weak Entity sets.

3. Transformation of Relationship sets.

4. Handling Composite Attributes.

5. Handling Multivalued Attributes.

6. Transformation of Generalization.

(1) Transformation of strong Entity Sets :

By the defination, the strong entity set has a primary key. thus it is very simple to cenvert a strong entity set to tables.

    (4) Handling Composite Attribute :

    We handle composite attributes by creating a seperate attribute for each of the component attributes : we do not create a seperate column for the composite attribute. Suppose Customer Address is a composite attribute of entity set customer and the component of customer address are street, city and state. The table generated from customer would then contain columns street, city and state.
