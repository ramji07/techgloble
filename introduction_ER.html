Introduction-ER :

The Entity Relationship Model (E-R Model) was introduced in a key article by Chen (1976) in which he described the main constructs of the E-R model - entities & relationships and their associated attributes.

An Entity Relationship Model is a detailed logical representation of the data for an organisation or for a business area. An E-R Model is normally expressed as an Entity Relationship Diagram (or E-R Diagram), which is a graphical representation of an E-R Model.

The E-R Model is used to construct conceptual data model, which is representation of the structure and constraints of a database that is independent of the software and its associated data model that will be used to implement database.

The Entity Relationship Model is based on preception of real world (data) that consists of a set of basic objects called entities and of relationships among these objects.

Components of an E-R Model :

In this section we will make ourselves familiar with the components of the E-R model and their representation in the E-R diagrams. The E-R model forms the basis of the ERDS. The ERD represents the conceptual view of the database. The ERDS represent three main components entities, attributes and relationships.

(1) Entity :

An entity is a person, place, object, event or concept in the real world that is distinguishable from all other objects. An entity has a set of properties and the values for some set of

properties may uniquely identify an entity. An entity is represented by a set of attributes. For each attribute, there is a set of permitted values called the domain or value set of that attribute. Some examples of each of these types of entities are as follows:

Person : Employee, Student, Patient.

Place : City, State, Country.

Object : Machine, Building, Automobile.

Event : Sale, Registration, Renewal.

Concept : Account, Course, Work Center. Entity Set :

(2) Entity Set :

An entity set is a set of entities of the same type that share the same properties or attributes. Some examples of Entity Set are as follows :

  • Set of all persons who are customer of a given bank, can be defined as entity set customer.
  • Loan of all types given by a particular bank, can be defined as entity set loan.

    • (3) Strong Entity :

    A strong entity set is one that exists independent of other entity sets. In other words, an entity set that has a primary key is termed as strong entity set.

    (4) Weak Entity:

    A weak entity is an entity whose existence depends on some other entity set. In other words, an entity set which may not have sufficient attributes to form a primary key is termed as weak entity set.

    (2) Attributes :

    An attribute can be simply defined as a property or characterstic of an entity. Each entity set has a set of attributes associated with it. Following are some examples of common entity sets with attributes :

  • Studentb: Student ID , Student Name, Address, Phone Number.
  • Employee : Employee ID, Employee Name, Designation, Branch.
  • Account : Account Number, Account type, Balance.

    • Attributes used in the E-R model can be characterstic by following attribute type :

    (1) Simple & Composite Attributes:

    A simple (or atomic) attribute is an attributes that cannot broken into smaller subparts. For example- Attribute student ID cannot be divided into subparts.

    A composite attribute is an attribute that can be further broken down into smaller parts, For example, Attribute student Name can be divided in three subparts First Name, Middle Name, Last Name.

    (2) Single Valued & Multivalued Attributes:

    An attribute is said to be single valued attribute can have only one value. Example- Account Balance can have only one value at a time. This is an example of single valued attribute.

    An attribute is said to be multivalued attribute if it can have more than one value. For example, a student entity can have multiple value for the hobby attribute such as reading, music, movies and so on.

    (3) Stored & Derived Attribute :

    An attribute which is already present as an attribute for an entity is an stored or base attribute.

    An attribute which is derived from a stored attribute as it was not present as an attribute for an entity is said to be derived attribute. The value for this type of attribute can be derived from the values of other related attributes or entities.

    (4) Null Attribute :

    An attribute that can have null value is an null attribute. A null value is used when an entity does not have a value for an attribute. An unkonwn value may be either missing or not known.

    For example -an attribute Phone Number of an entity employee may or may not have value. All employee not necessarily have phone. Here phone number is a null attribute.

    (3) Relation (or Table) :

    The terms Relation & Table can be used interchangably. Each relation consist of a set of named columns. An attribute is a named column of a relation. Each row of a relation corresponds to a record that contains data attribute values for single entity.

    A relation has the following properties -

    1. In any given column of a table, all items are of the same kind whereas items in different columns may not be of the same kind.

    2. For a row, each column must have an atomic (indivisible) value and also for a row, a column cannot have more than one value.

    3. All rows of a relation are distinct. That is, a relation does not contain two rows which are identical in every column. That is, each row of the relation can be uniquely identified by its contents.

    4. The ordering of rows within a relation is immaterial. That is, we cannot retrieve any things by saying that from row number 5, column name is to be accessed. There is no order maintained for rows inside a relation.

    5. The columns of a relation are assigned distinct names and the ordering of these columns is immaterial.
